# 프로젝트 초점 (설계 정렬 메모)

작성 기준: 팀 설계 보고서 v1.0 (2026-04-13) 및 레포 스켈레톤과의 정합성을 맞추기 위한 요약입니다.

## 한 줄 요약

**경쟁사·외부 기술 주장을 검증하는 것을 넘어, CCU(및 이에 준하는 탄소·에너지 전환) 도입이 장기적으로 회사에 이득이 될지, 특히 기술도입 가능성·경제성 관점에서 조사하는 시스템**을 지향합니다.

예시 주제: SK 이노베이션 맥락에서 **DAC(직접공기포집)** 도입 타당성, (탄소 포집 기술 도입의 장기 경제성 등).

## 기존 스켈레톤과 맞는 부분

- 사내 사업 DB 대신 **DART 사업보고서(로컬 PDF)** 로 “당사 맥락”을 가정.
- 입력: 기술명 또는 **기사/질의 컨텍스트** (예: “우리 회사에 OO 기술 사업화·도입 가능성 평가”).
- **Scientific / Industrial / Regulatory** 병렬 → **Cross Validator** → **Report**.
- 외부 에이전트 결과는 이후 **메모리/캐시**에 두고 보고서에서 참조하는 구조로 확장 예정.
- 최종 보고서는 **TRL / MRL / CRI**를 쓰되, **주된 메시지는 기술도입 가능성** + (향후) **회사 정보 기반 로드맵·체크리스트**.

## 지금 단계에서의 범위 (팀 합의)

- 전 에이전트가 아니라 **우선 2개 에이전트(예: Scientific + Regulatory)** 부터 구현·연동하는 전제로 코드를 나눔.
- 나머지(Industrial, 교차검증 고도화, RAG·캐시)는 스키마·오케스트레이터에 자리만 두고 점진 연결.

## 규제 에이전트 (Regulatory) — 설계 §6.4 정렬

- **규제 원문 자체에 대한 GRADE식 “신뢰도”는 하지 않음** (공식 문서를 사실로 둠).
- 할 일은 **적용 가능성**: 인센티브 해당 여부(예: IRA 45Q), 충돌·리스크, 시점·수출·인증 등.
- 구현된 파이프라인 (`backend/app/agents/regulatory/`): **① Tavily** → **② OpenAI로 법령·정책명 추출** → **③ 포털** (국가법령정보센터 OC 키 시 `law.go.kr`, 미국 Federal Register 공개 API, EU는 Tavily의 eur-lex 스니펫 대체) → **④ OpenAI로 보수적 해석** (`reason`, `verdict`, `confidence` 등). 키가 없으면 단계 생략·폴백.

## 보고서 (Report Generator)

- 설계 §9 구조(Executive Summary, 클레임별, 논문/산업/규제, TRL·MRL·CRI, Future Work)를 목표로 하되, 스켈레톤은 **마크다운 + 도입 체크리스트**부터 확장.
- **Future Work / 자사 적용**: DART·RAG와 결합한 “울산 플랜트 폐열” 등 시너지 시나리오는 데이터·입력 방식이 정해지면 연결.

## 참고

- 상세 표·GRADE/CRAAP 가중치·테스트 케이스(DAC 기사)는 팀 **설계 보고서 전문**을 기준으로 합니다.
